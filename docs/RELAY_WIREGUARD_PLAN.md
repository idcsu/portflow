# 双节点加密中转实施边界

## 当前已完成

- 中转规则记录入口节点和出口节点，且两者必须不同；
- 创建、修改或删除中转规则时，入口与出口节点的配置版本在同一事务中递增；
- 两端 Agent 都会收到相同的停用规则，并能安全保存；
- WireGuard 尚未就绪时，控制 API 和 Web 不允许启用中转规则；
- Agent 已支持 TCP、UDP 和 TCP+UDP 双端代理：入口拨号到出口隧道地址，出口仅在该地址的内部端口监听；
- 节点通过心跳上报隧道地址，控制面分别跟踪入口和出口要求的配置版本；
- 当前行为不安装软件、不创建网络接口、不启动服务，也不修改防火墙。

## 建议的数据路径

```text
客户端
  -> 入口 Agent 公网监听端口
  -> WireGuard 加密隧道
  -> 出口 Agent 仅绑定隧道地址的内部监听端口
  -> 目标服务
```

WireGuard 只承担入口与出口之间的加密传输。目标拨号仍由出口 Agent 完成，因此不需要开启 Linux `ip_forward`，也不需要在出口节点配置 NAT 或全局策略路由。这能减少对现有网络服务的影响。

## 实际联调前需要确认的系统操作

以下操作目前均未执行，执行前必须逐项获得用户明确同意：

1. 确认各节点是否已经安装 WireGuard；缺少时才安装对应系统软件包。
2. 创建专用接口，例如 `pfwg0`，并选择与现有网段不冲突的专用地址段。
3. 为 WireGuard 选择一个 UDP 监听端口。
4. 如现有防火墙默认拒绝入站，只添加该 WireGuard UDP 端口的精确放行规则；不修改默认策略和其他规则。
5. 将接口配置和新增防火墙规则分别备份，并记录实际命令输出。

## 恢复方式

具体命令必须根据目标发行版和防火墙实现确认，原则上的恢复顺序是：

1. 在面板停用所有相关中转规则；
2. 停止并禁用 `pfwg0`；
3. 删除本次新增的 WireGuard UDP 放行规则；
4. 删除或移走本次创建的 WireGuard 配置文件；
5. 恢复操作前保存的网络与防火墙配置；
6. 验证原有监听端口、路由和服务状态。

不会使用清空防火墙、覆盖整个规则集或修改默认策略的命令。若目标机器的网段、发行版或防火墙实现尚未确认，不执行实际网络变更。
