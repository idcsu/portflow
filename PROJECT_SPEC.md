# 多节点端口转发面板：项目需求与技术基准

> 状态：初始需求基线  
> 用途：本文件是设计、开发、测试和验收的方向性依据。实现过程中如需改变核心范围、架构或安全边界，应先与用户确认并同步更新本文件。

## 1. 项目目标

开发一套稳定优先、面向内部运维使用的多节点端口转发管理系统。

系统应允许用户通过完整、美观的 Web UI 管理多个公网服务器节点，创建和维护 TCP/UDP 转发线路，并查看节点状态、连接情况、实时速率、累计流量、异常日志与操作审计。

预期使用规模：

- 使用者：2～3 人；
- 节点：初期约 5～6 个，设计时可按 10 个左右留有余量；
- 转发规则：数量可能明显多于节点，但整体属于小规模内部系统；
- 节点环境：均有公网 IP；
- 使用性质：自用运维、团队内部使用，不做商业计费平台。

稳定性是最高优先级。系统不仅要正常运行，还应尽量做到单个管理组件短暂故障时不影响已有转发，并能在异常后自动恢复。

## 2. 第一版转发能力

### 2.1 普通转发

数据路径：

```text
用户 -> 入口节点:监听端口 -> 目标地址:目标端口
```

适用于入口节点能够直接访问目标服务的情况。这种模式路径短、延迟低，默认优先推荐。

### 2.2 双节点中转

数据路径：

```text
用户 -> 入口节点:监听端口 -> 加密通道 -> 出口节点 -> 目标地址:目标端口
```

适用于需要改善网络路径、隔离或隐藏源站，或者只有出口节点能够访问目标服务的情况。

第一版最多支持入口、出口两个转发节点，不实现任意多跳。数据模型和接口设计可为未来多跳保留合理扩展空间，但不得因此显著增加首版复杂度。

### 2.3 协议范围

第一版支持：

- TCP；
- UDP；
- TCP + UDP 同端口组合规则。

HTTP、HTTPS、WebSocket 可以作为普通 TCP 流量透传。第一版不包含应用层反向代理、自动证书签发、请求头修改等网站网关功能。

## 3. 总体架构

系统由以下部分构成：

```text
浏览器 -> HTTPS -> 控制面后端 -> PostgreSQL
                         |
                  认证的长连接
                         |
                    节点 Go Agent
                         |
              TCP/UDP 转发与节点间加密通道
```

### 3.1 控制面

控制面负责：

- 用户登录和权限；
- 节点注册与管理；
- 转发线路配置；
- 配置版本与下发；
- 节点状态和统计汇总；
- 审计、日志和告警展示；
- 向 Web 前端提供 REST API 和实时状态接口。

控制面不得承载实际业务转发流量，避免成为性能瓶颈或单点故障源。

### 3.2 节点 Agent

每个节点运行一个轻量 Go Agent。Agent 主动连接控制面，从而不需要向公网暴露单独的 Agent 管理接口。

Agent 负责：

- 节点注册、认证、心跳和状态上报；
- 接收、校验和应用转发配置；
- 执行 TCP/UDP 转发；
- 管理双节点之间的加密连接；
- 上报连接数、速率、累计流量和错误；
- 在控制面暂时不可用时继续运行最后一份有效配置；
- 重启后从本地配置恢复已有转发。

### 3.3 节点间加密通道

双节点中转使用成熟、经过广泛验证的加密方案，优先考虑 WireGuard，不自行设计密码算法或私有加密协议。

实现阶段需要评估 WireGuard 与系统部署、动态配置、权限要求和跨平台目标是否匹配。若需要调整底层方案，必须仍满足“成熟、安全、可维护、数据面独立”的原则，并先更新本文件。

### 3.4 管理面与数据面隔离

这是必须遵守的核心原则：

- 面板重启不应中断已经运行的转发；
- 数据库短暂不可用不应中断已经运行的转发；
- Agent 与面板暂时断开时，已有转发应继续工作；
- Agent 重启后应自动恢复最后一份有效配置；
- 业务流量不得绕行管理面板；
- 新配置应用失败时，应保留或回退到上一份有效配置。

## 4. 初步技术选型

稳定性、可维护性和较低部署复杂度优先：

- 控制面后端：Go；
- 节点 Agent：Go；
- Web 前端：React + TypeScript；
- UI 组件：在正式搭建前从 Ant Design、shadcn/ui 等成熟方案中确定；
- 图表：ECharts；
- 数据库：PostgreSQL；
- 控制 API：REST；
- 实时状态与 Agent 通信：当前使用 15 秒 HTTPS 心跳携带有界状态快照，控制面故障不会反向阻塞数据面；后续只有在规模和实时性确有需要时才评估长连接；
- HTTPS 入口：Caddy；
- 首选部署方式：Docker Compose；
- Agent 服务守护：Linux systemd；
- Agent 首版目标平台：Linux AMD64、Linux ARM64。

首版不引入 Kubernetes、微服务拆分、消息队列等与当前规模不匹配的复杂设施。Redis 首版默认不使用，只有出现明确需求时再引入。

## 5. Web UI 与主要页面

前端需要是完整、响应式、美观的运维面板，信息层次清晰，避免只堆砌传统后台表格。

### 5.1 登录页

- 用户名和密码登录；
- 登录失败限制与清晰提示；
- 可选双因素认证入口。

### 5.2 仪表盘

- 在线与离线节点数量；
- 正常与异常线路数量；
- 当前总连接数；
- 今日上传和下载流量；
- 最近 24 小时流量趋势；
- 最近异常与最近操作；
- 主要节点或线路的健康状态卡片。

### 5.3 节点管理

- 节点名称、地区、IP、架构和 Agent 版本；
- 在线状态和最后心跳；
- CPU、内存、系统负载、网络速率和磁盘状态；
- 节点注册令牌；
- 一键复制安装或注册命令；
- 禁用、恢复和重新连接等管理操作；
- Agent 升级可作为后续功能，首版不强制实现自动升级。

### 5.4 转发线路

- 线路名称、启停状态和健康状态；
- TCP、UDP 或 TCP + UDP；
- 入口节点和监听端口；
- 直连模式或可选出口节点；
- 目标主机和目标端口；
- 实时速率、累计流量和连接数；
- 带宽限制、连接数限制和来源 IP 访问规则；
- 创建、编辑、启停、复制和删除操作。

创建线路建议使用三步流程：

```text
选择协议和入口 -> 选择直连或出口节点 -> 填写目标、限制和访问规则
```

应用配置前应尽量检查：

- 入口端口是否被占用；
- 是否与已有规则冲突；
- 入口到目标或出口到目标是否可达；
- 入口和出口之间是否可达；
- UDP 配置是否可用；
- Agent 是否有能力应用该配置。

失败时应展示具体失败节点和阶段，而不是笼统提示“启动失败”。

### 5.5 监控与日志

- 节点和线路的流量趋势；
- 当前与历史连接统计；
- Agent 运行日志；
- 转发启动、停止和失败原因；
- 登录记录与配置变更审计；
- 合理的日志轮转与数据保留策略。

### 5.6 系统设置

- 用户与角色；
- Agent 注册令牌；
- 节点身份和密钥吊销；
- 监控、日志和审计数据保留时间；
- 登录安全设置；
- 系统版本信息。

## 6. 用户与权限

首版采用简单角色模型：

- 管理员：用户管理、系统设置、节点注册、线路和审计的完整权限；
- 普通成员：按设计授予节点查看及线路日常管理权限，不允许修改关键安全设置。

不实现实名、客户租户、套餐、余额和商业计费系统。

## 7. 安全与防滥用

虽然是内部系统，仍应包含必要的安全边界：

- 密码安全存储，不保存明文密码；
- 可选双因素认证；
- 登录失败频率限制；
- Agent 使用独立身份与安全认证；
- 注册令牌一次性或短期有效；
- Agent 身份和密钥可吊销；
- 控制链路加密；
- 来源 IP 允许与拒绝规则；
- 目标地址黑名单与白名单能力；
- 单线路带宽和连接数限制；
- 创建、修改、启停、删除、登录和安全设置变更均记录审计；
- 默认限制普通成员转发到回环地址、链路本地地址、云元数据地址等敏感目标，管理员可按明确策略调整；
- Agent 只接受已认证控制端下发且通过校验的配置；
- 严格验证地址、端口、协议、限速数值等所有输入。

所谓“流量封禁”首版主要指 IP 访问控制、目标限制、连接数限制和带宽限制，不进行流量内容审查。

## 8. 稳定性和性能要求

### 8.1 基本原则

- 单节点直连默认优先，路径更短、性能更好；
- 仅在路由、隔离或访问条件确有需要时使用双节点中转；
- 转发性能主要受节点带宽、网络质量、目标服务和 CPU 影响，面板不得进入流量路径；
- 双节点中转会增加一跳网络延迟与加密开销，应在 UI 中清晰表达路径。

### 8.2 必须考虑的可靠性措施

- Agent 断线自动重连，并采用合理退避；
- 转发任务异常后自动恢复，同时防止无限快速重启；
- Agent 本地原子保存最后一份有效配置；
- 主机或 Agent 重启后自动恢复线路；
- 配置包含版本号，避免乱序和重复应用；
- 新配置应用失败时自动回退；
- 正确设置连接、握手、空闲与读写超时；
- 及时清理死连接、半开连接和 UDP 会话状态；
- 有界缓冲和背压控制，避免高流量导致内存无限增长；
- 每条线路可设置最大连接数和带宽上限；
- Agent 故障不能连带拖垮控制面或其他节点；
- 日志轮转和监控数据清理，避免磁盘写满；
- systemd 守护 Agent，并设置合理的重启策略；
- 优雅关闭，在合理时间内处理或终止现有连接；
- 状态上报与实际转发解耦，监控链路阻塞不得阻塞业务流量。

### 8.3 故障定位能力

系统应尽量帮助区分以下原因：

- 入口节点负载或内存异常；
- 节点带宽跑满；
- 节点间延迟或丢包；
- 目标服务器响应缓慢或不可达；
- 连接数量过多；
- 入口端口冲突；
- Agent 或加密通道异常；
- 配置下发或应用失败。

## 12. 当前机器的操作约束（重要）

以下约束适用于整个开发过程，优先级高，不得擅自绕过：

1. **默认不得修改当前机器的防火墙；确实必要时必须事先获得用户明确同意。**
   - 未获得明确同意前，不执行或修改 `iptables`、`ip6tables`、`nftables`、`firewalld`、`ufw` 等防火墙规则或配置；
   - 不得仅以测试方便、临时部署或端口开放为由擅自修改，应优先提供不修改防火墙的替代方案；
   - 如果某项必要功能确实无法绕开防火墙变更，必须在执行前向用户说明：变更理由、准备执行的具体命令或规则、涉及的端口和协议、对现有服务的可能影响，以及为何没有合适的替代方案；
   - 请求许可时必须同时提供完整的恢复方案，包括需要备份的现有规则、准确的回滚命令、恢复后的验证方式和异常时的处理办法；
   - 只有在用户针对该次具体变更明确同意后才能执行，授权不得自动扩大到其他机器、其他规则或后续变更；
   - 执行前应在不泄露敏感信息的前提下保存必要的当前状态，执行后立即验证现有服务和网络可用性；如结果异常，应按照事先说明的方案恢复并向用户报告；
   - 即使获得许可，也不得清空、重置或整体替换现有防火墙规则，只能进行范围最小、可追踪、可回滚的必要修改，除非用户明确批准了更大范围的操作。

2. **安装任何必要工具、软件包或系统依赖前，必须先获得用户明确同意。**
   - 包括但不限于通过 `apt`、`apt-get`、`yum`、`dnf`、`apk`、`snap`、`pip`、`npm -g`、`cargo install`、`go install` 等方式向当前机器安装工具；
   - 应先说明需要安装什么、用途、可能影响和替代方案，再请求许可；
   - 未获同意时，只能使用机器已有工具或采用不需要安装的方案。

3. 不得擅自改动机器上已有服务、监听端口、服务绑定、系统网络配置或正在使用的运行环境。

4. 测试应优先使用非特权、高位、动态分配或明确未占用的本地端口；测试结束后清理本项目启动的进程和临时资源。

5. 即使未来系统具备管理远程节点网络能力，也不得默认将当前开发机器作为可随意修改网络配置的测试节点。

## 13. 开发决策准则

后续遇到未明确的细节时，按以下顺序取舍：

1. 不影响已有转发；
2. 稳定和可恢复；
3. 安全且不自行发明密码协议；
4. 容易部署、观察和排障；
5. 保持架构简单；
6. 性能满足当前规模并保留合理余量；
7. 界面美观、操作清楚；
8. 最后才考虑非必要功能扩展。

任何会让数据面依赖控制面实时存活的设计、无界资源使用、无法回退的配置下发，或为首版引入明显过度复杂度的方案，均应避免。

首版代码范围已经完成。以下项目属于首版之外的增强能力，或必须在目标环境完成的外部验收，不计为代码实现缺口：

- WireGuard 接口、密钥与节点对等配置的自动化管理（基于既有隧道的双节点数据面已完成）；
- 历史连接明细分析，以及需要主动探测目标的网络延迟/丢包质量指标（当前只保存每个节点最近一次实时快照）；

- systemd Agent 在实际独立 Linux 节点上的安装、重启恢复和长时运行验收（需要部署方提供目标节点）。

当前 Agent 支持通过已存在的 WireGuard 私网执行双节点中转；若启用规则缺少有效隧道地址、出口地址与本机配置不匹配，或配置值超出安全边界，Agent 会拒绝整份新配置并继续使用上一份有效配置，避免产生虚假的成功状态。带宽上限按整条规则的双向合计值执行，TCP、UDP、所有连接和两个方向共享；双节点模式只在入口 Agent 整形一次。
